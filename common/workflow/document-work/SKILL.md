# Document Work — Obsidian 지식 허브에 정리 (스택 중립)

> 목적: 완료한 작업을 **사용자가 나중에 AI에게 물으면 바로 답이 나오는** 개인 지식 허브(Obsidian)에 쌓는다.
> 레포 md(코드 핸드오프, repo 커밋)와 **역할이 다르다** — 이건 사용자 개인의 크로스 프로젝트 요약 허브. 둘 다 유지.

## 1. 언제
의미 있는 단위를 마쳤을 때: 기능 구현 완료·버그 원인규명·아키텍처 결정·리서치 정리 등.
사소한 편집(오타·한 줄 수정)은 대상 아님. 작업 마무리 시 **"vault에 정리할까요?"를 먼저 챙긴다**(무인 자동실행 아님 — 완전 자동화는 hook 필요).

## 2. 어디에 (vault 자동 감지)
경로를 하드코딩하지 말고 감지한다. 없으면 **조용히 건너뛴다**(다른 PC·미설치 사용자에게 무해).
```bash
# macOS
cat "$HOME/Library/Application Support/obsidian/obsidian.json" 2>/dev/null   # .vaults[].path
# Linux:   ~/.config/obsidian/obsidian.json   /  Windows: %APPDATA%\obsidian\obsidian.json
```
- vault 내 **프로젝트/주제별 폴더**에 넣는다. 없으면 폴더 신설.
- 폴더마다 **MOC**(라우팅 진입점) 유지 — 노트가 많아져도 AI가 MOC만 먼저 읽고 관련 노트로 간다.

## 3. 무엇을 (자기완결 요약)
코드 세부가 아니라 **"뭘/왜 했고 지금 상태가 뭔지"**. 한 노트만 읽어도 답이 나오게.
frontmatter 유지:
```md
---
title: <제목>
type: analysis | decision | feature | research
status: <현재 상태>
date: YYYY-MM-DD
tags: [project/<이름>, ...]
---
```
- 새 사실이 생기면 **MOC 요약을 먼저 갱신**하고, 상세는 개별 노트에.

## 4. vault 저장 기준 (넣을 것 / 뺄 것)
- ✅ 프로젝트·사업·개인 고유 맥락, **내가 리서치한 데이터를 사용자 판단/프로젝트로 가공한 것**(→ 자산).
- ❌ 날것의 일반 지식(규제 원문·라이브러리 문서)·"지금 최신" 외부 사실 → vault에 안 넣음(낡음). 필요하면 그때 질의/웹검색.
- 리서치 기반 정리물은 **날짜·출처**를 남기고, 오래되면 "갱신할까요?"를 먼저 제안.

## 5. 마무리
어느 폴더의 어떤 노트를 추가/갱신했는지 경로로 보고. MOC 링크가 걸렸는지 확인.
