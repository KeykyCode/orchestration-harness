# AI/LLM Engineering (프레임워크 중립)

> 원칙은 프레임워크와 무관하게 항상 참. 프레임워크(LangGraph·Vercel AI SDK·LlamaIndex·직접구현)는 **예시**일 뿐 — 현 프로젝트가 쓰는 것에 맞춘다.

## 1. eval 먼저 (변경 전 안전망)
- 프롬프트·체인·리트리버를 바꾸기 **전에** 골든셋(대표 입력 + 기대 동작/근거)과 평가법이 있는지 확인. 없으면 최소본부터 만든다.
- 변경 **후 반드시 회귀 평가** → 이전 대비 개선/악화를 **수치**로. 지표: 정확도·근거율(citation)·실패율·지연·토큰비용. "좋아진 것 같다" 금지.
- 실패 케이스는 따로 모아 원인 분석(검색 실패인가 / 생성 실패인가).

## 2. RAG (근거가 답보다 먼저)
- 답이 틀리면 **리트리브 결과부터** 본다 — "문서에 답이 있었나"를 먼저 확인(검색 실패 vs 생성 실패 분리).
- **grounding 강제** — 컨텍스트 밖 내용은 지어내지 않게 프롬프트에서 명시. 가능하면 출처(문서 ID/청크)를 답에 남긴다.
- 청킹·임베딩 모델·top_k·리랭커는 **추측 말고 eval로 비교**. 한 번에 하나만 바꿔 측정.

## 3. 프롬프트를 코드처럼
- 코드 곳곳에 하드코딩해 흩뿌리지 말고 **한 곳(파일/모듈)에 모으고 버전**을 남긴다.
- 구조화 출력이 필요하면 **스키마 명시 + 파싱 실패 폴백**.
- 무작정 늘리지 않는다 — 토큰↑ = 비용·지연·주의분산↑. 짧고 명확하게.

## 4. 에이전트/그래프 오케스트레이션
- **상태(State) 스키마를 먼저** 확정. 노드는 순수하게, 부수효과는 경계로 몰아넣는다.
- 긴 워크플로는 **체크포인트·재시도** 설계 + **종료 조건**(무한 루프·비용 폭주 방지, 최대 반복/토큰 캡).
- 스트리밍이면 어느 노드에서 토큰이 흘러나오는지 명확히.
- 프레임워크 예: LangGraph(State+노드/엣지) · Vercel AI SDK(`streamText`/tool) · LlamaIndex · 직접구현. 개념은 동일.

## 5. 모델·비용·지연
- 단계별 **적정 모델 배치** — 라우팅·분류는 가벼운 모델, 최종 생성은 강한 모델. 무거운 모델을 기본으로 깔지 않는다.
- 컨텍스트 길이·토큰 수의 트레이드오프를 항상 의식. (Claude 모델·가격·캐싱은 `claude-api` 스킬 참조.)

## 6. 관측 가능성
- 트레이싱(LangSmith·자체 로그 등)이 있으면 실패를 **재현 가능하게** 남긴다.
- 디버깅 시 "왜 이 답이 나왔나"를 **리트리브 결과·프롬프트 입력까지 거슬러** 추적한다.

## 검증·기록
- 완료 = eval 회귀 통과 + 지표 전후 비교 기록. `.tasks/in-progress.md`에 `무엇 바꿈 / 지표 전후 / 남은 리스크` 한 줄.
- AI 로직이 백엔드·프론트에 얹히는 접점은 **입출력 계약만 정의**하고 구현은 `developer`에 (PM 경유) 넘긴다.
